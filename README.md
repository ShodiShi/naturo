# ShinobiLang

## Описание языка

`ShinobiLang` — учебный язык программирования, разработанный в рамках работы по дисциплине «Методы компиляции». Язык оформлен в тематике вселенной Naruto, однако по своей структуре является формальным алгоритмическим языком, предназначенным для демонстрации принципов лексического анализа, синтаксического анализа, генерации ОПС и интерпретации.

Ниже приведено описание базовых возможностей языка, соответствующих первым четырём обязательным требованиям задания.

## 1. Типы данных

В базовой версии языка `ShinobiLang` используются следующие типы данных и программные объекты:

- целочисленные константы;
- переменные без явного объявления типа;
- одномерные статические массивы.

Числовые константы записываются как последовательности десятичных цифр.

Примеры:

```txt
0
5
123
```

Переменные в языке не требуют предварительного описания типа. Имя переменной задаётся идентификатором, состоящим из латинских букв, цифр и символа `_`, причём первый символ не может быть цифрой.

Примеры идентификаторов:

```txt
chakra
itachi
power_1
```

Для хранения наборов однотипных значений в языке предусмотрены одномерные статические массивы. Массив объявляется оператором `scroll`, а его размер задаётся при объявлении и остаётся фиксированным в течение выполнения программы.

Пример объявления массива:

```txt
scroll squad[5];
```

В данной записи:

- `scroll` — ключевое слово объявления массива;
- `squad` — имя массива;
- `5` — фиксированный размер массива.

Доступ к элементу массива осуществляется по индексу в квадратных скобках:

```txt
squad[0]
squad[1]
```

## 2. Операторы присваивания и формулы

Для присваивания в языке используется оператор `:=`.

Общий вид оператора присваивания:

```txt
variable := expression;
```

Примеры:

```txt
chakra := 10;
power := chakra + 5;
squad[0] := 7;
```

Язык поддерживает арифметические выражения со скобками и операциями двух приоритетов:

- низший приоритет: `+`, `-`;
- высший приоритет: `*`, `/`.

При вычислении выражений сначала выполняются операции более высокого приоритета, затем более низкого. Скобки позволяют явно задавать порядок вычислений.

Примеры выражений:

```txt
power := chakra + 2 * 3;
power := (chakra + 2) * 3;
result := chakra - 4 / 2;
```

Во втором примере сначала вычисляется сумма `chakra + 2`, а затем результат умножается на `3`.

## 3. Условные операторы и циклы

Язык `ShinobiLang` поддерживает условные операторы и циклы с условиями. Условия строятся на основе операций сравнения:

- `==` — равно;
- `!=` — не равно;
- `<` — меньше;
- `<=` — меньше либо равно;
- `>` — больше;
- `>=` — больше либо равно.

### Условный оператор

Для условного выполнения команд используется оператор `ifjutsu`.

Общий вид:

```txt
ifjutsu condition then
    statement_list
end;
```

Пример:

```txt
ifjutsu chakra > 10 then
    show chakra;
end;
```

Если условие истинно, выполняются операторы между `then` и `end`.

### Цикл

Для организации цикла с предусловием используется оператор `train`.

Общий вид:

```txt
train condition do
    statement_list
end;
```

Пример:

```txt
train chakra > 0 do
    chakra := chakra - 1;
end;
```

В данном примере тело цикла выполняется до тех пор, пока значение переменной `chakra` больше нуля.

## 4. Операторы ввода и вывода

Для ввода данных в языке используется оператор `sense`, а для вывода — оператор `show`.

### Оператор ввода

Общий вид:

```txt
sense variable;
```

Примеры:

```txt
sense chakra;
sense squad[0];
```

Оператор `sense` позволяет вводить значения как в обычные переменные, так и в элементы массива.

### Оператор вывода

Общий вид:

```txt
show expression;
```

Примеры:

```txt
show chakra;
show chakra + 5;
show squad[1];
```

Оператор `show` выводит значение переменной, элемента массива или произвольного арифметического выражения.

## Пример программы

Ниже приведён пример программы на языке `ShinobiLang`, демонстрирующий использование переменных, арифметических выражений, цикла и вывода:

```txt
sense chakra;
total := 0;

train chakra > 0 do
    total := total + chakra;
    chakra := chakra - 1;
end;

show total;
```

Данная программа считывает значение переменной `chakra`, затем последовательно суммирует числа от `chakra` до `1` и выводит итоговый результат.

## 5. Список лексем

Для языка `ShinobiLang` определён фиксированный набор лексем. Каждая лексема представляет собой минимальную значимую единицу исходного текста, распознаваемую лексическим анализатором.

В языке используются следующие основные группы лексем:

### 5.1. Идентификаторы и константы

- `ID` — идентификатор переменной или массива;
- `NUMBER` — целочисленная константа.

Идентификатор задаётся последовательностью латинских букв, цифр и символа `_`, при этом первый символ не может быть цифрой.

### 5.2. Ключевые слова

В языке используются следующие служебные слова:

- `SENSE` — оператор ввода `sense`;
- `SHOW` — оператор вывода `show`;
- `IFJUTSU` — условный оператор `ifjutsu`;
- `THEN` — начало тела условного оператора `then`;
- `TRAIN` — цикл с предусловием `train`;
- `DO` — начало тела цикла `do`;
- `END` — конец блока `end`;
- `SCROLL` — объявление массива `scroll`.

### 5.3. Арифметические операции

- `PLUS` — символ `+`;
- `MINUS` — символ `-`;
- `MUL` — символ `*`;
- `DIV` — символ `/`.

### 5.4. Операции сравнения

- `EQ` — `==`;
- `NE` — `!=`;
- `LT` — `<`;
- `LE` — `<=`;
- `GT` — `>`;
- `GE` — `>=`.

### 5.5. Разделители и скобки

- `ASSIGN` — `:=`;
- `LPAREN` — `(`;
- `RPAREN` — `)`;
- `LBRACKET` — `[`;
- `RBRACKET` — `]`;
- `SEMICOLON` — `;`;
- `EOF` — конец входного файла.

Полная таблица лексем с номерами, примерами и назначением приведена в файле `Kompilyator.xlsx` на листе `Лексемы`.

## 6. Лексический анализатор

Лексический анализатор предназначен для преобразования исходного текста программы в последовательность лексем, пригодную для дальнейшего синтаксического анализа.

Входом лексического анализатора является текст программы на языке `ShinobiLang`, а выходом — упорядоченный список токенов. Каждый токен содержит:

- тип лексемы;
- текст лексемы;
- номер строки;
- номер позиции в строке.

Лексический анализатор в проекте должен быть реализован как **табличный детерминированный конечный автомат**.

### 6.1. Принцип работы

Работа лексического анализатора строится следующим образом:

1. исходный текст считывается посимвольно;
2. каждый символ относится к определённому классу входных символов;
3. по текущему состоянию автомата и классу входного символа из таблицы переходов выбирается следующее состояние;
4. если переход существует, символ включается в формируемую лексему;
5. если переход отсутствует, текущая лексема считается завершённой;
6. на основе достигнутого состояния создаётся соответствующий токен либо фиксируется ошибка.

Таким образом, лексический анализатор не кодирует все варианты разбора вручную через длинные цепочки `if`, а использует таблицу переходов автомата.

### 6.2. Классы входных символов

Для упрощения описания автомата все символы разбиваются на классы:

- буква или `_`;
- цифра;
- `:`;
- `=`;
- `!`;
- `<`;
- `>`;
- `+`;
- `-`;
- `*`;
- `/`;
- `(`;
- `)`;
- `[`;
- `]`;
- `;`;
- пробельные символы;
- конец файла;
- прочие недопустимые символы.

Такое разбиение позволяет не рассматривать каждый конкретный символ отдельно, а работать с группами символов, которые одинаково влияют на поведение автомата.

### 6.3. Состояния автомата

В лексическом анализаторе используются начальное, промежуточные и финальные состояния.

Примеры состояний:

- начальное состояние `S0`;
- состояние распознавания идентификатора `S1`;
- состояние распознавания числа `S2`;
- состояние после чтения `:` для распознавания `:=`;
- состояние после чтения `=` для распознавания `==`;
- состояние после чтения `!` для распознавания `!=`;
- состояния для распознавания `<`, `<=`, `>`, `>=`.

Также отдельные финальные состояния соответствуют лексемам `+`, `-`, `*`, `/`, `(`, `)`, `[`, `]`, `;` и `EOF`.

Полный список состояний автомата приведён в файле `Kompilyator.xlsx` на листе `Таблица переходов`.

### 6.4. Распознавание идентификаторов и ключевых слов

При чтении последовательности символов вида буква/цифра/`_` автомат переходит в состояние распознавания идентификатора.

После завершения чтения такой последовательности выполняется дополнительная проверка:

- если полученный текст совпадает с одним из ключевых слов языка (`sense`, `show`, `ifjutsu`, `then`, `train`, `do`, `end`, `scroll`), то создаётся токен соответствующего ключевого слова;
- иначе создаётся токен `ID`.

### 6.5. Распознавание составных операций

Некоторые лексемы состоят из двух символов. К ним относятся:

- `:=`;
- `==`;
- `!=`;
- `<=`;
- `>=`.

Для их распознавания автомат использует промежуточные состояния. Например, после чтения символа `:` автомат переходит в состояние ожидания `=`, а при чтении символа `<` — в состояние, где определяется, является ли лексема `<` или `<=`.

### 6.6. Обработка ошибок

Если для текущего состояния и класса входного символа переход в таблице отсутствует, а текущее состояние не является завершающим, фиксируется лексическая ошибка.

Лексическая ошибка должна содержать:

- номер строки;
- номер ошибочного символа в строке;
- текст сообщения об ошибке.

Пример сообщения:

```txt
Лексическая ошибка: строка 4, символ 7. Недопустимый символ '@'.
```

## 7. Табличное описание автомата

Для лексического анализатора языка `ShinobiLang` подготовлено табличное описание конечного автомата. Оно включает:

- список состояний автомата;
- список классов входных символов;
- таблицу переходов;
- список финальных состояний и соответствующих им лексем.

Все эти данные оформлены в файле `Kompilyator.xlsx` на листе `Таблица переходов`.

Такая форма представления соответствует требованиям задания, согласно которым лексический анализатор должен быть реализован как табличный автомат, а не как набор вручную закодированных частных случаев.

## 8. КС-грамматика языка

Для языка `ShinobiLang` построена контекстно-свободная грамматика (КС-грамматика), описывающая синтаксис допустимых программ.

КС-грамматика определяет:

- структуру программы в целом;
- допустимые типы операторов;
- правила записи присваивания;
- правила записи условных операторов и циклов;
- правила построения выражений;
- правила использования массивов и индексов;
- правила записи условий и операций сравнения.

Грамматика языка включает следующие основные нетерминалы:

- `program` — программа;
- `statement_list` — список операторов;
- `statement` — отдельный оператор;
- `assignment` — присваивание;
- `array_declaration` — объявление массива;
- `variable` — переменная или элемент массива;
- `if_statement` — условный оператор;
- `while_statement` — цикл;
- `sense_statement` — оператор ввода;
- `show_statement` — оператор вывода;
- `condition` — условие;
- `rel_op` — операция сравнения;
- `expression` — выражение;
- `expression_tail` — продолжение выражения;
- `term` — терм;
- `term_tail` — продолжение терма;
- `factor` — минимальная единица выражения.

КС-грамматика уже оформлена в файле `Kompilyator.xlsx` на листе `Грамматика`.

### 8.1. Назначение КС-грамматики

КС-грамматика используется как формальное описание языка и служит основой для построения синтаксического анализатора.

Именно по этой грамматике синтаксический анализатор будет определять:

- допустима ли последовательность лексем;
- к какой конструкции языка относится текущий фрагмент программы;
- какие семантические действия необходимо выполнить при разборе.

### 8.2. Особенности построения грамматики

При проектировании грамматики были учтены следующие требования:

- поддержка операторов присваивания, ввода, вывода, условия и цикла;
- поддержка одномерных статических массивов;
- поддержка арифметических выражений с двумя уровнями приоритета операций;
- возможность дальнейшего преобразования грамматики к форме, удобной для табличного синтаксического анализа.

Для выражений грамматика записана без левой рекурсии. Вместо правил вида:

```txt
expression -> expression PLUS term
```

используется разложение на:

```txt
expression -> term expression_tail
expression_tail -> PLUS term expression_tail | MINUS term expression_tail | ε
```

Такое представление удобнее для построения табличного парсера и последующего преобразования грамматики.

## 9. Синтаксический анализатор

Синтаксический анализатор предназначен для проверки правильности последовательности лексем, полученной от лексического анализатора, и для построения внутреннего представления программы.

В соответствии с требованиями задания синтаксический анализатор должен быть реализован как **магазинный автомат с табличным управлением**.

Это означает, что парсер не должен представлять собой просто набор вложенных `if` и `switch`, а должен работать на основе:

- стека;
- таблицы разбора;
- правил грамматики;
- семантических действий.

### 9.1. Нужно ли писать парсер вручную

Да, синтаксический анализатор должен быть написан самостоятельно. Однако писать его нужно не как произвольный набор проверок, а как реализацию табличного магазинного автомата по заранее заданной КС-грамматике.

Иными словами:

- грамматика задаёт правила языка;
- по грамматике строится таблица разбора;
- парсер использует стек и эту таблицу;
- при разборе выполняются семантические действия для генерации ОПС.

Таким образом, парсер пишется вручную, но на основе формальной модели, а не «в лоб».

### 9.2. Общий принцип работы парсера

Работа синтаксического анализатора организуется следующим образом:

1. на вход поступает последовательность лексем;
2. в стек помещается стартовый нетерминал грамматики;
3. рассматривается вершина стека и текущая входная лексема;
4. если на вершине стека терминал, выполняется его сопоставление с текущей лексемой;
5. если на вершине стека нетерминал, по таблице разбора выбирается нужное правило грамматики;
6. правая часть выбранного правила помещается в стек;
7. если на вершине стека находится семантическое действие, оно выполняется;
8. процесс продолжается до полного разбора входной программы.

### 9.3. Табличное управление

Таблица разбора представляет собой отображение вида:

```txt
(нетерминал, текущая лексема) -> правило грамматики
```

По этой таблице парсер выбирает, какое правило необходимо применить в текущей ситуации.

Например, если на вершине стека находится нетерминал `statement`, а текущая входная лексема равна `SENSE`, то из таблицы выбирается правило, соответствующее оператору ввода.

### 9.4. Связь парсера с генерацией ОПС

Синтаксический анализатор в проекте должен не только проверять синтаксис программы, но и одновременно формировать внутреннее представление в виде обратной польской записи.

Для этого в правила грамматики встраиваются семантические действия, которые вызываются в нужные моменты разбора.

Например, при разборе:

- числовой константы будет формироваться операция помещения числа в стек ОПС;
- переменной — операция загрузки значения;
- арифметической операции — соответствующая операция `ADD`, `SUB`, `MUL`, `DIV`;
- присваивания — операция сохранения результата;
- условия или цикла — операции условного и безусловного перехода.

### 9.5. Ошибки синтаксического анализа

Если текущая входная лексема не соответствует ожидаемой конструкции по таблице разбора, синтаксический анализатор должен выдавать сообщение об ошибке.

Синтаксическая ошибка должна содержать:

- номер строки;
- номер символа в строке;
- краткое описание ошибки.

Пример сообщения:

```txt
Синтаксическая ошибка: строка 5, символ 10. Ожидалась лексема SEMICOLON.
```

## 10. Связь грамматики и синтаксического анализатора

КС-грамматика и синтаксический анализатор тесно связаны между собой:

- грамматика формально задаёт язык;
- таблица разбора строится по грамматике;
- парсер реализует разбор входной программы на основе этой таблицы;
- семантические действия, встроенные в правила, позволяют одновременно генерировать ОПС.
